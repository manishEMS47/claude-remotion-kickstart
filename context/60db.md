# 60db.ai Audio API (TTS + STT)

60db provides both **text-to-speech** (voiceovers) and **speech-to-text**
(transcription) from a single API. In this project it is offered **alongside
ElevenLabs** (TTS) and **Deepgram** (STT) via a unified `--provider` switch on
the `/generate-voiceover` and `/transcribe` slash commands.

- **Base URL:** `https://api.60db.ai`
- **Auth header:** `Authorization: Bearer $SIXTYDB_API_KEY`
- All requests are JSON except STT, which is `multipart/form-data`.

---

## Text-to-Speech (synthesize)

**Endpoint:** `POST https://api.60db.ai/tts-synthesize`

Returns a single JSON payload with the full audio as base64 (best for
generating a complete voiceover file).

### Request body (JSON)

| Field           | Type    | Required | Default | Notes                                              |
| --------------- | ------- | -------- | ------- | -------------------------------------------------- |
| `text`          | string  | yes      | —       | Text to synthesize (max **5000** chars)            |
| `voice_id`      | string  | no       | system  | Voice identifier (see `GET /myvoices`)             |
| `voice`         | string  | no       | —       | Alternative voice name (instead of `voice_id`)     |
| `enhance`       | boolean | no       | `true`  | Audio quality improvement                          |
| `speed`         | number  | no       | `1`     | Playback rate, `0.5`–`2.0`                          |
| `stability`     | number  | no       | `50`    | `0`–`100` (lower = more expressive)                |
| `similarity`    | number  | no       | `75`    | `0`–`100` (closeness to source voice)              |
| `output_format` | string  | no       | `mp3`   | `mp3`, `wav`, `ogg`, `flac`                        |

### Response (JSON)

- `success` (boolean)
- `message` (string)
- `audio_base64` (string) — base64-encoded audio bytes
- `sample_rate` (number) — Hz
- `duration_seconds` (number)
- `encoding` (string)
- `output_format` (string)

### Example

```bash
curl -X POST https://api.60db.ai/tts-synthesize \
  -H "Authorization: Bearer $SIXTYDB_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"text":"Hello, world!","voice_id":"default-voice","output_format":"mp3"}'
```

Decode the audio to a file (Bash):

```bash
curl -sS -X POST https://api.60db.ai/tts-synthesize \
  -H "Authorization: Bearer $SIXTYDB_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"text":"Hello, world!"}' \
| jq -r '.audio_base64' | base64 -d > public/audio/voiceover.mp3
```

> On Windows where `base64`/`jq` may be missing, decode with PowerShell:
> `[IO.File]::WriteAllBytes("public/audio/voiceover.mp3", [Convert]::FromBase64String((Get-Content resp.json | ConvertFrom-Json).audio_base64))`

## Text-to-Speech (streaming)

**Endpoint:** `POST https://api.60db.ai/tts-stream`

Same body parameters as `/tts-synthesize` (minus `output_format`). Returns
**newline-delimited JSON (NDJSON)**, one message per line:

- `chunk` — `result.audioContent` holds a base64 audio chunk (concatenate in order)
- `complete` — stream finished successfully
- `error` — `message` describes the failure

```bash
curl -N -X POST https://api.60db.ai/tts-stream \
  -H "Authorization: Bearer $SIXTYDB_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"text":"Your text here","voice_id":"default-voice"}'
```

Use streaming only for low-latency playback. For rendering Remotion videos,
prefer `/tts-synthesize` (a single finished file is simpler to prefetch).

## Text-to-Speech (WebSocket realtime)

**URL:** `wss://api.60db.ai/ws/tts` (auth via query param: `?apiKey=$SIXTYDB_API_KEY`)

On connect the server sends `connection_established` (with `credit_balance`).

**Client → Server:** `create_context` → `send_text` → `flush_context` → `close_context`

```json
{ "create_context": { "context_id": "c1", "voice_id": "<id>",
  "audio_config": { "audio_encoding": "LINEAR16", "sample_rate_hertz": 24000 },
  "speed": 1, "stability": 50, "similarity": 75 } }
```

**Server → Client:** `context_created`, `audio_chunk` (`audioContent` base64),
`flush_completed`, `context_closed`, `error`.

| Encoding       | Sample rates              | Format                                   |
| -------------- | ------------------------- | ---------------------------------------- |
| `LINEAR16`/`PCM` | 8000, 16000, 24000, 48000 | Raw PCM, 16-bit signed LE, mono          |
| `MULAW`/`ULAW`   | 8000 only                 | G.711 µ-law, mono                        |
| `OGG_OPUS`       | 24000 only                | Ogg Opus (each chunk is a self-contained file) |

> LINEAR16/PCM/MULAW chunks concatenate directly; `OGG_OPUS` chunks must be
> handled as separate files.

## Voices (list my voices)

**Endpoint:** `GET https://api.60db.ai/myvoices` (no parameters)

### Response

- `success`, `message`
- `data` (array of voice objects):
  - `voice_id` (string, UUID) — use this as `voice_id` in TTS
  - `name` (string)
  - `category` — `"cloned"` | `"professional"`
  - `model` — `"60db Fast"` | `"60db Quality"`
  - `labels` — `{ language, language_name, gender, accent }`
    (accent: American | British | Indian | Neutral)
  - `description` (string | null)
  - `is_native` (boolean)
  - `available_for_tiers` (array)
  - `categories` (array of recommended use cases)

```bash
curl -sS https://api.60db.ai/myvoices \
  -H "Authorization: Bearer $SIXTYDB_API_KEY" | jq '.data[] | {voice_id, name, model}'
```

## Speech-to-Text (transcribe)

**Endpoint:** `POST https://api.60db.ai/stt` — `Content-Type: multipart/form-data`

### Fields

| Field                | Type    | Default | Notes                                            |
| -------------------- | ------- | ------- | ------------------------------------------------ |
| `file`               | file    | —       | **Required.** WAV/MP3/M4A/OGG/FLAC/WebM/MP4 audio. Max 10 MB / 1 hour |
| `language`           | string  | `auto`  | ISO 639-1 code or `auto`                         |
| `return_timestamps`  | string  | none    | Set to `word` for word-level timing              |
| `include_confidence` | boolean | `false` | Per-word confidence scores                       |
| `diarize`            | boolean | `false` | Speaker identification                           |
| `min_speakers` / `max_speakers` | integer | — | Diarization bounds                       |
| `context`            | string  | —       | Domain/speaker hint for LLM refinement           |
| `keywords`           | string  | —       | Custom vocabulary boost (CSV)                    |
| `languages`          | string  | —       | Constrain language candidates (CSV)              |
| `script_correction`  | boolean | `false` | Devanagari/Latin normalization                   |

For caption timing, **always pass `return_timestamps=word` and
`include_confidence=true`.**

### Response (JSON)

- `request_id`, `text` (full transcript), `language`, `language_name`, `duration_sec`
- `segments[]` — each has `start`, `end`, `text`, `confidence`, and:
  - `words[]` — `{ word, start, end, confidence?, boosted?, original? }`
  - `speakers[]` — `{ speaker, start, end }` (when `diarize=true`)
- `words` — flat word list across all segments
- `warnings`, `warning_codes`, `language_detection`

### Example

```bash
curl -X POST https://api.60db.ai/stt \
  -H "Authorization: Bearer $SIXTYDB_API_KEY" \
  -F "file=@recording.mp3" \
  -F "language=auto" \
  -F "return_timestamps=word" \
  -F "include_confidence=true"
```

## Mapping to the Remotion caption pipeline

60db STT word objects (`{ word, start, end, confidence }`) and Deepgram word
objects (`{ word, start, end, confidence, punctuated_word }`) are both
normalized by `src/utils/loadTranscript.ts` → `normalizeTranscript()` into a
single `TranscriptWord[]`. That feeds `segmentTranscript()` → `<Caption>`
unchanged, so transcripts from **either provider** drive captions identically.
